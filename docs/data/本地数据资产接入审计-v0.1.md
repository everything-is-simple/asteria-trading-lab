## 2. 字段映射（向项目标准对齐）

### 2.1 symbol（标的代码）

| 来源 | 形态 | 项目标准 (`ts_code`) |
|---|---|---|
| TDX vipdoc 文件名 | `sz000001.day` | `000001.SZ` |
| DuckDB `instrument_master.symbol` | `sz000001` | `000001.SZ` |
| DuckDB `instrument_master.exchange` | `SZ` / `SH` / `BJ` | （提取后缀） |
| Baostock `code` | `sz.000001` | `000001.SZ` |

**映射规则**: `<exchange-prefix><code> | <exchange>.<code>` → `<code>.<EXCHANGE>` （已在 `readers.py:_normalize_duckdb_symbol` / `_parse_symbol_from_path` 实现）

### 2.2 trade_date（交易日）

所有源使用 ISO `YYYY-MM-DD`，DuckDB 用 `DATE` 类型，TDX 二进制为 `YYYYMMDD` 整数（已在 `_format_yyyymmdd` 处理）。

### 2.3 daily_bar（日 K）

| 项目标准字段 | DuckDB `base_bar` | Baostock `query_history_k_data_plus` | TDX `.day` 二进制偏移 |
|---|---|---|---|
| ts_code | symbol | code | 文件名 |
| trade_date | trade_dt | date | 0-3 (int) |
| open | open | open | 4-7 (int / 100) |
| high | high | high | 8-11 (int / 100) |
| low | low | low | 12-15 (int / 100) |
| close | close | close | 16-19 (int / 100) |
| amount | amount | amount | 20-23 (float) |
| volume | volume | volume | 24-27 (int) |
| adjustment | analysis_price_line | adjustflag | （文件目录区分） |

### 2.4 复权语义（关键差异）

| 源 | 标识 | 含义 |
|---|---|---|
| DuckDB `analysis_price_line` | `backward` | 后复权（向后调整） |
| Baostock `adjustflag` | `1` | 后复权 |
| Baostock `adjustflag` | `2` | 前复权 |
| Baostock `adjustflag` | `3` | 不复权 |
| TDX 离线 | 目录 `Backward-Adjusted` / `Forward-Adjusted` / `Non-Adjusted` | 后 / 前 / 不复权 |

**⚠️ 已知差异**: 本审计的双源对账中，Baostock 用了 `adjustflag=3` (不复权)，DuckDB `base_bar` 取了 `analysis_price_line='backward'`（后复权）。两套数值仍然完全一致 —— 因为 2026-01 距离最近一次除权较近、调整因子≈1。**该差异必须在后续真实样本回填时按 `(ts_code, trade_date)` 区间严格对齐复权口径**，不能假设永远一致。

### 2.5 trading_calendar

| 项目标准 | DuckDB `trade_calendar` | Baostock `query_trade_dates` |
|---|---|---|
| exchange | exchange (`SH`/`SZ`) | （全市场统一） |
| trade_date | trade_dt | calendar_date |
| is_open | is_open | is_trading_day |

### 2.6 sector_membership（行业归属）

| 项目标准 | DuckDB `industry_block_relation` |
|---|---|
| ts_code | symbol |
| relation_type | relation_type (`industry`) |
| relation_code | relation_code (e.g. `T040202`) |
| relation_name | relation_name（GBK 编码，见已知问题 §4.1） |
| effective_from / effective_to | effective_from / effective_to |

**这是 GPT 今天做"行业标签时间对齐"的真正数据源** —— `industry_block_relation` 自带 `effective_from/effective_to`，`first_batch` 的样本窗口重叠匹配应直接走这张表，而不是任何"最新快照"。

### 2.7 制度事实（A 股制度审计层的天然底座）

| 项目标准 | DuckDB `tradability_fact` |
|---|---|
| ts_code | symbol |
| trade_date | trade_dt |

**这张表 19.2M 行就是 A 股制度事实的真值表** —— `AShareExecutionConstraintSnapshot` 的 `evidence_ref` 应该指向这里的 `(symbol, trade_dt, source_manifest_hash)` 三元组，而不是任何手工录入的清单。

**结论**: 双源**主体一致**，可作为 MALF 真实样本回填的数据底座。差异点已登记，需后续在跨复权区间样本中重新校验。

## 4. 已知问题（登记，不在本审计修复）

### 4.1 DuckDB 中文字段 GBK 编码

`instrument_master.name` / `industry_block_relation.relation_name` 在 DuckDB 中以 GBK 字节存储为 VARCHAR，直接读取出现乱码（如 `���շ��`）。`src/data_sources/tdx_local/readers.py` 当前未做 GBK→UTF-8 转换。

**影响**: 仅影响中文展示，不影响 `symbol / trade_date / OHLCV / relation_code / effective_from/to` 等所有结构化字段。MALF 链路目前不依赖中文名做判定，**不阻断**当前主线。

**待办**: 在下一份 PR 里给 `_normalize_duckdb_symbol` 旁边补一个 `_safe_decode_gbk_name(value)` 工具，跨表统一处理。

如 §2.4 / §3 所述，Baostock `adjustflag` 与 DuckDB `analysis_price_line` 标签语义不同，需要在跨区间样本中严格对齐。

**待办**: 在双源校验扩到 2-3 只股票 + 不同年份后，沉淀一份 `跨源复权口径对齐-v0.1.md`。
